# C 风格字符串与旧代码接口(c_style_strings_and_legacy_interfaces)

## 核心概念

- C 风格字符串本质上是以 `'\0'` 结尾的字符数组。
- 它们历史上很常见，但在现代 C++ 里通常不是首选文本表示。
- `<cstring>` 提供的是基于指针和结尾空字符的操作接口。
- 学这一部分的重点，是理解旧接口和现代类型如何互相配合。

## 什么才算 C 风格字符串

```cpp
char ca1[] = {'C', '+', '+'};
char ca2[] = "C++";
```

这两者最关键的区别不在于写法，而在于结尾是否有 `'\0'`。

- `ca1` 不是合法的 C 风格字符串
- `ca2` 才是，因为字符串字面值会自动补上结尾空字符

也可以把这条规则反过来记：

- 不是所有 `char` 数组都是字符串
- 只有以 `'\0'` 结尾的那种才是

## `<cstring>` 里的常用函数

```cpp
#include <cstring>
```

常见函数：

- `strlen(p)`：长度，不算 `'\0'`
- `strcmp(p1, p2)`：比较两个字符串
- `strcpy(p1, p2)`：拷贝
- `strcat(p1, p2)`：拼接

例如：

```cpp
const char* p = "hello";
auto len = strlen(p);
```

这些函数的共同特点是：它们相信你传入的是合法字符串，并且目标空间足够大。

## 为什么它更容易出问题

```cpp
char small[4] = "abc";
// strcat(small, "def");
```

这类接口的问题不是“函数不好用”，而是它把很多安全前提交给了调用者自己维护，比如：

- 是否真的有 `'\0'`
- 缓冲区是否足够大
- 指针是否指向合法内存

这就是现代 C++ 更偏向 `string` 的原因之一。

## 和现代类型的衔接

`string` 转成旧接口常用的字符指针：

```cpp
string s = "hello";
const char* p = s.c_str();
```

这里要记住：

- `c_str()` 返回的是 `const char*`
- 这个指针依赖原来的 `string`
- 一旦 `string` 内容变化，之前拿到的指针可能失效

数组初始化 `vector`：

```cpp
int ia[] = {0, 1, 2, 3, 4};
vector<int> nums(begin(ia), end(ia));
vector<int> part(ia + 1, ia + 4);
```

这部分的重点不是语法记忆，而是学会在现代容器和旧式数组之间切换。

## 易错点

- 把任意 `char` 数组都当成 C 风格字符串。
- 忘记 C 风格字符串依赖 `'\0'` 结尾。
- 误以为 `c_str()` 返回的指针可以长期脱离原 `string` 独立存在。
- 使用 `strcpy`、`strcat` 时不先确认目标空间是否足够。
