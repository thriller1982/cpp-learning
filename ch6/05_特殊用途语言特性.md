# 特殊用途语言特性(special_purpose_language_features)

## 核心概念

- 这一节的重点包括默认实参、`inline`、`constexpr`、`assert` 和 `NDEBUG`。
- 默认实参让你可以少传右侧那些已有默认值的参数。
- `inline` 更像是给编译器的建议，而不是强制命令。
- `constexpr` 函数既强调“有机会在编译期求值”，也天然带有 `inline` 属性。
- `assert` 适合调试期检查程序员假设，不是普通用户输入校验工具。

## 默认实参

函数参数可以提供默认值：

```cpp
string screen(size_t ht = 24, size_t wid = 80, char backgrnd = ' ');
```

调用时就可以少传右边那些参数：

```cpp
screen();
screen(30);
screen(30, 100);
```

默认实参最重要的规则是：通常应从右往左连续提供。  
也就是说，调用时你只能省略最右边连续的一段参数，不能跳着省。  
一旦某个形参有默认值，它右边的形参通常也都必须有默认值。
例如下面这种形式是自然的：

```cpp
void f(int a, int b, int c = 0);
```

而前面给了默认值、后面反而没有默认值的写法通常不合法，因为实参是按位置一个一个匹配的。

默认值也通常写在函数声明里，而不是在多个地方重复写，这样接口信息更集中。

## `inline`

`inline` 常用于体积小、调用频繁的函数：

```cpp
inline bool isShorter(const string& s1, const string& s2) {
    return s1.size() < s2.size();
}
```

它的含义不是“编译器一定会展开这个函数”，而是表明这是一个适合作为内联候选的小函数。  
是否真的展开，最终仍由编译器决定。

头文件中定义短小函数时，也经常会看到 `inline`。这是因为多个源文件如果都要用到它，使用处通常需要看到函数定义。  
不过这不等于 `inline` 只能写在头文件里；如果它只在一个 `.cpp` 里使用，放在这个 `.cpp` 中也可以。  
即使放在 `.cpp` 里，也不必非得写在所有函数最前面，关键是让编译器在使用前看到需要看到的声明或定义。

## `constexpr` 函数

`constexpr` 函数强调的是：如果条件满足，它有机会在编译期求值：

```cpp
constexpr int new_sz() {
    return 42;
}
```

如果函数体和调用实参都满足要求，它就可以用于常量表达式场景。

`constexpr` 函数本身也隐式是 `inline`，所以如果它要被多个源文件共用，通常也会放在头文件里。  
如果希望别处把它真正用于编译期求值，那么定义通常也需要在使用处可见。  
不过 `constexpr` 不等于一定在编译期执行；条件满足时它可以在编译期求值，不满足时也可能像普通函数一样在运行期调用。

因此：

- `inline` 更偏向小函数和可能的展开
- `constexpr` 更偏向编译期求值能力

它们不是同一类含义。

## `assert` 与 `NDEBUG`

`assert` 用于在调试时检查某个条件是否成立：

```cpp
assert(!s.empty());
```

如果条件为假，程序通常会报告问题并终止。  
它主要用来验证“这里本来就不该出错”的程序员假设。

如果定义了 `NDEBUG`，`assert` 通常会被关闭。  
因此 `assert` 更适合调试期检查，而不是普通用户输入校验：

- 用户输入可能出错，这是正常分支问题
- 某个内部前提本不该被破坏，这更像 `assert` 的场景

## 易错点

- 默认实参写得不连续。
- 把默认值分散写在多个地方，导致接口信息不统一。
- 误以为 `inline` 一定会强制展开，或者必须写在头文件最前面。
- 以为 `constexpr` 一定会在编译期执行，没有意识到它也可能在运行期调用。
- 用 `assert` 代替正常的用户输入校验逻辑。
