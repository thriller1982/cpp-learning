# 函数匹配(function_matching)

## 核心概念

- 名字查找先发生；函数匹配讨论的是同名重载已经找到之后，编译器如何选一个版本。
- 匹配要先找候选函数，再筛可行函数，最后比较哪个版本最好。
- 比较“好坏”时，关键看每个实参到形参需要什么转换。
- 常见情况下，精确匹配优于 `const` 转换，`const` 转换优于类型提升，类型提升优于一般算术或指针转换。
- 如果没有唯一最佳版本，调用就会产生二义性。

## 函数匹配在解决什么问题

前面讲重载时，重点是“同名函数能不能共存”；函数匹配讨论的是“这些同名函数已经找到之后，调用点到底该选哪一个”。

可以先把这个过程压成四步：

- 名字查找先找到同名函数
- 从中找出候选函数
- 再筛出可行函数
- 最后选出唯一最佳匹配

如果中途没有候选函数、没有可行函数，或者最后没有唯一最佳版本，调用都会报错。

## 候选函数

候选函数是名字查找之后得到的那组同名函数。

例如：

```cpp
void print(int);
void print(double);
void print(const string&);
```

当编译器在当前作用域里找到 `print` 这个名字时，这几个版本都会先进入候选函数集合。

但候选函数只是“可能被考虑”，并不等于最后一定能参与调用。  
真正能不能调，还要继续看实参数量和类型是不是匹配。

## 可行函数

可行函数是从候选函数里进一步筛出来的。  
一个重载要想成为可行函数，至少要满足两件事：

- 实参数量能和形参数量对应上
- 每个实参都能转换到对应形参类型

例如：

```cpp
void print(int);
void print(const string&);

string s = "hello";
print(s);
```

这里两个 `print` 都是候选函数，但 `print(int)` 不能拿 `string` 实参来调用，所以它不是可行函数；真正可行的是 `print(const string&)`。

引用形参能不能绑定上，也属于可行性的判断范围。  
例如一个 `const` 对象不能绑定到普通引用形参上，这种重载会先被淘汰，而不是留到后面再比较。

如果候选函数里一个可行函数都没有，编译器就会报“没有匹配的函数”。

## 寻找最佳匹配

当可行函数不止一个时，编译器才开始比较：哪个版本更好。

它比较的不是“哪个看起来差不多能调”，而是每个实参到对应形参所需的转换。  
某个候选只有在下面这种情况下，才可能成为最佳匹配：

- 至少有一个实参，它比别的候选更好
- 其他实参上，它又不比别的候选更差

例如：

```cpp
void f(int);
void f(double);

f(42);
```

这里两个版本都可行，但 `42` 到 `int` 是精确匹配，到 `double` 需要算术转换，所以 `f(int)` 更好。

因此匹配不是“先找到一个能用的就停”，而是“把所有可行函数放在一起比较，最后必须选出唯一最佳版本”。

## 匹配中的常见转换等级

初学阶段最常见的几层，可以先按“从好到差”这样记：

**1. 精确匹配**

实参类型和形参类型正好一致，通常是最好的情况。

```cpp
void f(int);
void f(double);

f(42);   // 选 f(int)
```

这里 `42` 本身就是 `int`，所以 `f(int)` 比 `f(double)` 更直接。

**2. `const` 转换**

如果只是为了匹配形参而增加 `const` 限定，这一层通常比精确匹配稍弱。

```cpp
void look(string&);
void look(const string&);

string s = "hi";
look(s);   // 更倾向选 look(string&)
```

对于普通 `string` 对象，两个版本都可能可行，但不加 `const` 的引用绑定得更直接。  
如果实参本身是 `const string`，那普通引用版本就不再可行，只能选 `const string&`。

**3. 类型提升**

像 `char`、`short` 这类较小整型，传参时经常会先提升成 `int`。  
这种提升通常比一般算术转换更自然。

```cpp
void g(int);
void g(long);

short s = 1;
g(s);   // 通常选 g(int)
```

这里 `short -> int` 属于类型提升，而 `short -> long` 更像普通算术转换，所以前者更好。

**4. 算术转换或指针转换**

再往下一层，就是更一般的算术转换或指针转换。  
它们往往还能让调用成立，但匹配层级更弱。

```cpp
void h(long);

h(3.14);   // double -> long，属于算术转换
```

像某些指针类型之间的普通转换，也可以落在这一层。  
只要更高层的匹配同时存在，这类转换通常不会是优先选择。

更复杂的类类型转换、用户自定义转换和模板相关规则，会让函数匹配再复杂一层；这一节先不展开。

## 二义性

如果多个可行函数一样好，编译器就无法选出唯一最佳版本，这时调用会产生二义性错误。

最常见的情况是：两个版本都要经过差不多层级的普通转换，但谁也不比谁更好。

```cpp
void f(long);
void f(double);

// f(42);
```

这里 `42` 转成 `long` 可以，转成 `double` 也可以。  
但这两个转换都不是精确匹配，也没有哪一个明显更优，所以调用会产生二义性。

多参数时也会出现类似问题：一个候选在第一个参数上更好，另一个候选在第二个参数上更好，结果整体上依然没有唯一最佳版本。

所以二义性的关键不是“都不能调”，而是“都能调，但无法唯一地决定该选谁”。

## 重载设计的提醒

函数匹配看起来复杂，很多时候不是因为规则本身太可怕，而是因为重载设计得太接近。

如果你写出一组重载，让调用经常依赖隐式转换来猜，那么调用点就会越来越不直观。  
尤其要警惕下面这些情况：

- 参数类型过于接近
- 常见调用经常要靠转换才能命中
- 不仔细比较就不知道会选哪个版本

比起让调用者记住一堆隐式规则，更好的做法通常是把接口意图写得更明确，尽量减少“并列第一”的重载组合。

## 易错点

- 以为名字相同的重载都会自动参与最后的调用选择。
- 把候选函数、可行函数和最佳匹配混成同一层。
- 以为只要某个重载“能调”，编译器就一定能把它选出来。
- 忽略类型提升和一般算术转换不是同一层级。
- 看到重载调用报错，却没意识到可能是二义性。
