# 函数匹配(function_matching)

## 核心概念

- 重载函数调用时，编译器必须选出唯一且最合适的版本。
- 精确匹配通常优于需要转换的匹配。
- 少转换通常优于多转换。
- 如果多个候选同样好，调用就会产生二义性。
- 重载不是“能调就行”，而是“必须唯一且最佳”。

## 候选与可行函数

一旦存在多个同名重载函数，调用点就会出现一个新问题：到底该选哪一个版本。  
编译器在这里做的事，就是函数匹配。

它不是随便挑一个“差不多能用的”，而是要先找出候选函数，再从中筛出可行函数，并最终选出唯一最佳版本。

因此函数匹配关注的不只是“能不能调用”，还包括“是否能选出唯一最合适的那个”。

## 精确匹配与类型转换

精确匹配通常优于需要转换的匹配。例如：

```cpp
void f(int);
void f(double);

f(42);
```

这里 `42` 本身就是 `int`，所以 `f(int)` 通常是更自然的选择。

如果多个候选都能调用，编译器还会比较转换代价。  
一般来说，需要的转换越少、越直接，匹配就越好。

这也是为什么参数类型彼此很接近的重载，往往更容易让调用点变得不直观。

## 二义性

如果多个候选一样好，编译器就无法选出唯一最佳版本，这时调用会产生二义性错误。

例如：

```cpp
void f(long);
void f(double);

// f(42);
```

这里 `42` 转成 `long` 和转成 `double` 都可行，但编译器不一定能判断哪个更优。

因此“能转过去”还不够，还必须能选出唯一最佳版本。

## 重载设计

函数匹配之所以会复杂，通常不是因为重载本身难，而是因为类型转换规则开始介入。

设计重载接口时，应尽量避免下面这些情况：

- 参数类型过于接近
- 很多调用都依赖隐式转换
- 不仔细看就不知道会选中哪个版本

比起让匹配规则变得晦涩，更好的做法通常是让调用意图本身更明确。

## 易错点

- 以为只要某个重载“能用”，编译器就一定能选出来。
- 忽略精确匹配和类型转换在匹配中的优先级差异。
- 看到重载调用报错，却没意识到可能是二义性。
- 写出一组参数类型非常接近的重载，把调用者逼进隐式转换中。
- 把“函数重载”和“函数匹配”当成一回事，而没意识到前者定义接口、后者选择接口。
