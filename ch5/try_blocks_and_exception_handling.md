# try 语句块与异常处理(try_blocks_and_exception_handling)

## 核心概念

- 异常机制用来把正常逻辑和错误处理逻辑分开。
- `throw` 负责抛出异常，`try` 负责包住可能出错的代码，`catch` 负责处理异常。
- 一旦抛出异常，当前正常流程会被打断，程序会去找匹配的处理代码。
- 标准库已经提供了一批常见异常类型。
- 异常不是普通分支语句，不应该拿来替代日常条件判断。

## 为什么要有异常处理

程序执行过程中，有些问题不是“走另一条普通分支”就能解决的。  
例如：

- 输入数据不合法
- 两个对象状态不匹配
- 某个操作根本无法继续

这时如果把错误处理和正常逻辑全揉在一起，代码会越来越乱。  
异常机制的价值，就是把“正常路径”和“错误路径”拆开。

## `throw` 表达式

当程序发现某种异常情况时，可以直接抛出异常：

```cpp
throw runtime_error("input error");
```

它的含义是：

- 当前正常流程不再继续
- 把这个错误对象抛出去
- 交给后续匹配的异常处理代码

这不是“返回一个特殊值”，而是明确中断当前普通执行路径。

## `try` 和 `catch`

异常处理最基本的结构是：

```cpp
try {
    // 可能出错的代码
} catch (const runtime_error& err) {
    // 处理这种异常
}
```

可以先这样理解：

- `try` 里放可能失败的代码
- 一旦抛出异常，就停止当前正常流程
- 程序会去找后面的 `catch`
- 找到类型匹配的 `catch`，就进入处理逻辑

例如：

```cpp
try {
    if (item1.isbn() != item2.isbn())
        throw runtime_error("Data must refer to same ISBN");
    cout << item1 + item2 << endl;
} catch (const runtime_error& err) {
    cout << err.what() << endl;
}
```

这里最值得抓住的不是细节语法，而是流程：

- 正常时就继续往下执行
- 出错时就离开正常路径，转去错误处理代码

## 异常处理的基本流程

可以把它记成三步：

1. `throw` 抛出异常对象
2. `try` 包住可能出错的代码
3. `catch` 按类型接住并处理

如果当前地方没有匹配的 `catch`，异常就会继续往外层传。  
如果一路都没有处理，程序通常就无法正常继续。

## 标准异常类型

标准库已经提供了一些常见异常类型，例如：

- `exception`
- `runtime_error`
- `logic_error`
- `out_of_range`

复习阶段先抓住两个点就够了：

- `runtime_error` 很常见，适合表示运行时才发现的问题
- 很多异常对象都可以通过 `what()` 取出错误信息

例如：

```cpp
catch (const runtime_error& err) {
    cout << err.what() << endl;
}
```

## 异常不是普通控制流替代品

异常确实能改变程序路径，但它不适合拿来替代普通的 `if` 判断。  
更准确地说：

- 正常、可预期的分支，用条件语句处理
- 真正的异常情况，再考虑用异常机制

这样代码的职责边界才清楚。

## 易错点

- 把 `throw` 当成“返回错误码”的另一种写法。
- 以为抛出异常后，`try` 里后面的语句还会继续正常执行。
- `catch` 住异常后，却不知道可以用 `what()` 看错误信息。
- 把异常机制当成普通分支结构来替代 `if` / `else`。
- 只记住 `throw`、`try`、`catch` 三个关键词，却没抓住它们对应的流程关系。
